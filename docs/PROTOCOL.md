## Overview

If you do not know Minecraft protocol details (and/or do not want AI slop), this is the short version:

1. A Minecraft client connects to your HAProxy server.
2. The client sends a small handshake packet first ("hello, i'm a client").
3. HAMineGate reads only this first packet and checks a few things:
    - is the packet valid?
    - what hostname did the player use?
    - is this a status ping (in multiplayer screen) or a real login attempt (when client clicks Connect)?
4. If the packet is broken or suspicious, HAProxy rejects the connection early.
5. If the packet is valid but hostname is not on your allowlist, HAProxy rejects it.
6. If checks pass, HAProxy forwards the same TCP connection to your real backend (Velocity).

So basically, `HAMineGate` is a gatekeeper. It inspects the first Minecraft packet, blocks bad or unwanted traffic, and only lets approved connections reach your backend server.

### Why check hostnames?

In short, hostname checking lets you implement a simple allowlist (ACL in `haproxy.cfg`) without extra backend logic. See the main [README.md](../README.md#why-implement-a-hostname-allowlist) for my reasoning.

This document explains how HAMineGate processes inbound Minecraft Java Edition TCP connections before forwarding them to the backend.

## Scope

Protocol handling for routing and filtering is implemented in:

- `src/minecraft_prot.lua` (HAProxy Lua action)
- `src/haproxy.cfg` (frontend policy and ACL decisions)

> [!NOTE]
> The logic targets the Minecraft **handshake** packet only (the first packet sent by a Java client).

## High-Level Flow

1. Client opens TCP connection to HAProxy frontend `minecraft` on port `25565`.
2. HAProxy waits up to `500ms` (`tcp-request inspect-delay 500ms`) to collect initial bytes.
3. HAProxy runs Lua action `lua.mc_handshake`.
4. Lua decodes and validates handshake fields:
    - packet length (VarInt)
    - packet id (must be `0x00`)
    - protocol version (VarInt)
    - server address / hostname (String)
    - server port (unsigned short, skipped)
    - next state (VarInt, must be `1` or `2`)
5. Lua writes parsed results into transaction variables:
    - `txn.mc_proto`
    - `txn.mc_host`
    - `txn.mc_state`
6. HAProxy ACLs use these variables to reject invalid or unauthorised requests, otherwise route to backend `velocity`.

## Handshake Packet Layout

Minecraft Java handshake (packet id `0x00`) is interpreted as:

```txt
VarInt packet_length
VarInt packet_id
VarInt protocol_version
String server_address
Unsigned Short server_port
VarInt next_state
```

Where:

- `next_state = 1` means Status (server list ping path, ie what is seen in client multiplayer screen).
- `next_state = 2` means Login path.

## Decoder (`minecraft_prot.lua`)

### Reader model

The parser uses a payload cursor represented as:

- `payload[1]`: raw byte string
- `payload[2]`: 1-based read index

Helpers advance the cursor as values are decoded.

### VarInt handling

`payload_read_varint(payload, max_bytes, nilable)`:

- Decodes Minecraft-style VarInt.
- Returns:
  - decoded integer on success
  - `nil` if data is incomplete and `nilable == true`
  - `-1` on parse failure (missing required bytes or too many bytes)

### String handling

`payload_read_string(payload, max_prefix_bytes, max_utf8_len)`:

- Reads VarInt length prefix.
- Enforces maximum length and available bytes.
- Returns decoded string, or `false` for invalid input.

### Validation and fail-fast rules

The handshake is rejected (`false`) if any check fails:

- packet length invalid (`-1`) or too large (`> 4096`)
- packet id is not `0`
- protocol version invalid (`-1` or `<= 0`)
- hostname string invalid
- not enough bytes for 2-byte port
- next state not in `{1, 2}`

If data is only incomplete (not invalid), parser returns `nil` so HAProxy can retry when more bytes arrive.

### Hostname normalization

After reading `server_address`, the parser strips any suffix after a NUL byte (`\0`).

This handles modern forwarding patterns where extra metadata may be appended to hostname-like fields.

## Return Semantics

`read_mc_handshake` returns one of three states:

- `nil`: not enough data yet
- `false`: invalid/illegal handshake
- `true, proto, host, state`: valid handshake

The HAProxy action `mc_handshake(txn)` maps those states to txn vars:

- On `nil`: no vars are updated (HAProxy may call action again)
- On `false`:
  - `txn.mc_proto = 0`
  - `txn.mc_host = ""`
  - `txn.mc_state = 0`
- On success:
  - `txn.mc_proto = <protocol_version>`
  - `txn.mc_host = <normalized_hostname>`
  - `txn.mc_state = <1|2>`

## HAProxy Policy Layer (`haproxy.cfg`)

After Lua action runs:

1. Reject malformed/failed parse:

    ```haproxy
    tcp-request content reject if { var(txn.mc_proto) -m int 0 }
    ```

2. Host allowlist ACLs:

    ```haproxy
    acl allowed1 var(txn.mc_host) -m reg ^([^.]+\.)*mc\.qincai\.xyz\.?$
    acl allowed3 var(txn.mc_host) -m str 2407:7000:f030:c684::1
    tcp-request content reject if !allowed1 !allowed3
    ```

3. If accepted, forward to backend `velocity`.

## Why This Works Reliably

- Incomplete input is handled without prematurely rejecting valid clients.
- Invalid packets are fast-failed early to reduce backend exposure.
- Parsing is strict enough to stop malformed traffic but lightweight enough for TCP inspect stage.
- Routing decisions are made from explicit decoded fields, not raw pattern matching on opaque bytes.
